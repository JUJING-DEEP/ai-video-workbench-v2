# v1.1 Release Gates

Target release: `v1.1`
Theme: Jimeng Provider Contract Hardening Release

## v1.1 Must Have

- Provider contract tests for the existing Jimeng lifecycle:
  - settings validation
  - submit job
  - poll job
  - download result
  - create video asset
  - bind video to shot
  - reject invalid terminal-state polling
  - preserve credential redaction
- Real Jimeng feasibility decision documented before release.
- Mock Jimeng lifecycle remains deterministic and green in default tests.
- Failed, timed-out, or rejected jobs do not bind video outputs.
- Credential values never appear in public API responses or frontend prefill
  state.
- Real-provider tests, if added, are skipped unless explicitly enabled.

## v1.1 Should Have

- Additional deterministic demo or provider-job fixtures that do not require
  external provider credentials.
- README and SPEC updates that clearly describe the real Jimeng support status.
- Manual validation notes for one real Jimeng job if the feasibility gate passes.
- Focused provider/job test files if existing tests become too large to review.

## v1.1 Out of Scope

- Connecting real Jimeng in PR 1.
- New providers.
- New public API endpoints.
- New frontend pages.
- New database tables.
- FFmpeg final rendering.
- Subtitle product workflow beyond existing subtitle file generation.
- Voice-over workflow.
- Browser agents or account automation.
- Background queues, cloud rendering, multi-user features, or production hosting.

## Real Jimeng Feasibility Gate

This gate decides whether real Jimeng can be included in a later v1.1 PR. It
does not connect the real provider in PR 1.

Required evidence:

- Official endpoint, region, model, request schema, response schema, polling
  schema, and download behavior are identified.
- Account access and model permissions are confirmed.
- Authentication and signing requirements are understood.
- Cost, quota, rate limits, and expected generation limits are documented.
- At least one short controlled real job succeeds before any release claiming
  real Jimeng support.
- Redacted request and response examples are captured if real access is
  available.

Exit decision:

- Pass: continue with a narrow real integration behind the existing `jimeng`
  provider contract.
- Partial: keep real Jimeng experimental and opt-in only.
- Fail: release v1.1 as provider contract hardening and defer real Jimeng.

## Provider Contract Test Gate

Default tests must cover:

- successful submit stores `submit_id` and status `submitted`
- processing poll keeps status `processing`
- completed poll downloads output, creates a `video` asset, binds the shot, and
  marks the job `completed`
- provider failure marks the job `failed`
- timeout marks the job `timeout`
- terminal jobs cannot be polled again
- jobs without `submit_id` cannot be polled
- failed, timed-out, and rejected jobs do not bind output paths
- public responses do not contain credential values

The gate passes only when contract tests run without real credentials.

## Security Gate

Required checks:

- Provider settings GET responses expose credential booleans only.
- Provider settings PUT responses expose credential booleans only.
- Frontend credential inputs are not prefilled from saved settings.
- Logs, fixtures, and docs do not include real secrets.
- Real-provider tests use environment variables and are skipped by default.
- Local plaintext credential storage remains documented as a trusted-local
  limitation unless replaced by a scoped secret-storage decision.

## CI Gate

Default CI must stay free of real provider calls and paid external jobs.

Required checks:

- Backend tests pass.
- Frontend tests pass.
- Frontend build passes.
- Backend lint passes.
- Frontend lint passes.
- Real Jimeng integration tests are skipped unless explicit opt-in credentials
  and flags are present.

## Demo Validation Gate

Required checks:

- Coffee Demo remains self-contained.
- Import Demo still creates the project, imports storyboard text, registers demo
  assets, binds shot paths, and saves timeline order.
- Render-plan generation still follows saved timeline order.
- Exported render plan contains the expected video paths and shot ordering.
- Any new v1.1 fixture is deterministic and does not require real provider
  credentials.
