# v1.1 Roadmap

Date: 2026-06-30
Repository: JUJING-DEEP/ai-video-workbench
Baseline: `main` / `v1.0.0`

## Planning Inputs

- `v1.0.0` is published on `origin/main` at `55fba46`.
- `v1.0.0` shipped a self-contained Coffee Demo, timeline-ordered render-plan export, credential redaction, Jimeng REST Adapter mock job submit/poll/download scaffolding, and smoke coverage.
- The current Jimeng REST Adapter remains a workflow-testing mock. It does not prove real Volcano Engine Jimeng API compatibility.
- Volcano Engine official documentation currently lists Jimeng video generation API references, including [video generation 3.0 Pro](https://www.volcengine.com/docs/85621/1777001), [720P](https://www.volcengine.com/docs/85621/1792704), and [1080P](https://www.volcengine.com/docs/85621/1792711) endpoints. v1.1 should still treat real API availability as a gated integration risk until credentials, account permissions, signing, quota, and cost behavior are verified with a controlled account.

## v1.1 Goal

Make the existing Jimeng video workflow production-shaped and testable without expanding the product surface: define and harden the provider contract, verify whether the real Jimeng API can be integrated through the existing Jimeng provider path, and improve demo/test confidence around provider-driven video jobs.

v1.1 should not attempt to become a full rendering, subtitle-production, voice-over, or agent release.

## Scope

### Must Have

- Provider contract tests for the existing video provider lifecycle:
  - settings validation
  - submit job
  - poll processing/completed/failed/timeout states
  - result download
  - asset creation and shot binding
  - terminal-state protections
  - no credential leakage in API responses or logs
- Real Jimeng integration feasibility spike against the existing `jimeng` provider identity, not a new provider:
  - confirm official endpoint, model name, auth/signature requirements, request shape, result shape, polling shape, and download mechanism
  - document whether the real API is available to the project account
  - capture a small redacted request/response contract fixture if access is available
- Provider credential security hardening plan and tests:
  - keep public responses secret-free
  - avoid logging secrets
  - document local plaintext SQLite storage as a trusted-local v1.1 limitation unless OS/env secret storage is explicitly chosen for v1.1
- Job polling reliability:
  - verify state transitions with contract tests
  - define polling interval/backoff guidance
  - define max poll duration and timeout state behavior
- Video result download hardening:
  - validate result URL expectations
  - enforce download timeout and size limits
  - write downloaded output through existing project storage conventions
  - preserve asset library and shot binding behavior
- Failure and timeout behavior:
  - map provider auth errors, quota errors, validation errors, network errors, provider failures, download failures, and timeouts to stable local job statuses/messages
  - ensure failed jobs do not bind partial or missing videos
- Cost control:
  - keep real-provider tests opt-in
  - require explicit environment variables for integration tests
  - limit test prompts, duration, resolution, and job count
  - document expected cost risk before enabling real calls

### Should Have

- Additional demo fixtures that exercise current capabilities without external providers:
  - one longer multi-shot fixture with mixed Mode B and key-node video shots
  - one failure-oriented fixture for missing video/keyframe/timeline validation
  - one provider-job fixture using mocked Jimeng job states
- Split oversized provider/job tests into focused files if needed to keep v1.1 changes reviewable.
- README/SPEC updates that clarify:
  - real Jimeng support status
  - how to run mock contract tests
  - how to run opt-in real Jimeng integration tests
  - credential and cost warnings

### Could Have

- A manual v1.1 validation checklist for one real Jimeng job from keyframe to downloaded video to render-plan export.
- A dry-run mode for provider contract validation if it can be implemented without new API surface.
- Minimal storage cleanup for failed or timed-out downloaded artifacts if needed by the provider job implementation.

### Out of Scope

- New providers.
- New public API endpoints.
- New frontend pages.
- New database tables.
- Browser agents or account automation.
- FFmpeg final rendering as a shipped v1.1 feature.
- Subtitle workflow beyond existing subtitle file generation.
- Voice-over workflow.
- End-to-end autonomous production workflow.
- Multi-user, cloud, queue, or background worker architecture.

## Feature Placement Decisions

### Real Jimeng Provider

Recommended for v1.1 only if implemented as hardening of the existing `jimeng` provider path. Do not add a second Jimeng provider or new provider selector option.

Shipping real Jimeng as v1.1 requires all release criteria below. If official account access, signing, quota, or download semantics cannot be verified early, v1.1 should ship provider contract tests and documentation while keeping real Jimeng marked experimental or deferred.

### Provider Contract Tests

Recommended as the core v1.1 deliverable. These tests reduce risk before any real provider work and protect v1.0 behavior while Jimeng changes.

### Additional Demo Fixtures

Recommended for v1.1 as Should Have. They are useful because they improve confidence without requiring external API calls. Keep fixtures small and deterministic.

### FFmpeg Final Rendering

Recommend deferring to v1.2. v1.1 should keep render-plan export as the boundary and avoid combining provider-network risk with final media-rendering risk.

### Subtitle Workflow

Recommend deferring to v1.2. Existing subtitle file generation can remain, but a product workflow for subtitle review, alignment, burn-in, and export belongs with FFmpeg final rendering.

### Voice-over Workflow

Recommend deferring to v1.2 or later. Voice-over introduces provider/tool choice, audio timing, transcript alignment, cost, and asset lifecycle concerns that should not be mixed into v1.1.

### Agent Workflow

Recommend deferring to v2.0. Agents introduce browser/session state, login/captcha handling, external UI drift, interruption/resume semantics, and a larger trust boundary. v1.1 should focus on deterministic provider contracts first.

## Milestones

### Milestone 1: v1.1 Contract Baseline

- Freeze the existing Jimeng mock lifecycle as an explicit provider contract.
- Add focused contract tests for submit, poll, download, bind, terminal states, provider errors, timeout errors, and credential redaction.
- Confirm v1.0 demo and render-plan behavior still pass.

### Milestone 2: Real Jimeng Feasibility Gate

- Verify official documentation, account availability, endpoint/model names, request signing, quota, pricing, and result download behavior.
- Decide one of:
  - proceed with real Jimeng implementation in v1.1
  - keep real Jimeng behind opt-in experimental tests
  - defer real Jimeng to v1.2 if access or contract remains uncertain

### Milestone 3: Provider Reliability Hardening

- Implement or document polling interval, timeout, and retry policy.
- Harden result download handling.
- Keep all real-provider tests opt-in and cost-bounded.
- Ensure failures produce useful local job state without binding bad outputs.

### Milestone 4: Demo And Documentation Confidence

- Add deterministic demo fixtures if still valuable after contract tests.
- Update README/SPEC/release notes for v1.1 provider status, credential model, and test commands.

## Risks

### Real Jimeng API Availability

Risk: Official docs exist, but project account access, model permissions, signature requirements, region support, quota, and result download behavior may not match assumptions.

Mitigation: Run the feasibility gate before implementation. Do not promise real Jimeng in v1.1 release notes until a real account submits, polls, downloads, and binds at least one short controlled job.

### Provider Credential Security

Risk: Credentials are redacted from responses, but local SQLite plaintext storage remains a trust-boundary limitation.

Mitigation: Keep v1.1 local-only unless secret storage changes. Add/keep regression tests for no credential echo, no prefill, and no secret logging. Document the limitation prominently.

### Job Polling

Risk: Real provider states may not map cleanly to current `submitted`, `processing`, `completed`, `failed`, and `timeout` states.

Mitigation: Define a provider-state mapping table in tests/docs. Unknown remote states should fail safely or remain processing with a bounded timeout.

### Downloading Video Results

Risk: Result URLs may expire, require signed requests, redirect, be large, or fail after a completed job state.

Mitigation: Treat download as its own failure point. Enforce size/time limits and do not mark a local job completed until the file is written, asset-created, and shot-bound.

### Failed Retry

Risk: Blind retries can duplicate paid jobs or bind stale outputs.

Mitigation: v1.1 should not auto-retry paid submissions by default. Prefer explicit manual retry guidance and idempotent local handling. If retry is added, limit it to safe network/download retry classes.

### Cost Control

Risk: Contract/integration tests can spend money or consume quota.

Mitigation: Make real-provider tests opt-in with environment variables, mark them separately, cap job count to one short sample, and exclude them from default CI.

### Timeout Handling

Risk: Provider requests, long-running jobs, and downloads can hang or leave ambiguous local state.

Mitigation: Define separate submit, poll, total-job, and download timeouts. Persist timeout state with clear messages.

## Test Plan

- Default backend tests:
  - provider settings redaction and validation
  - Jimeng mock contract submit/poll/download/bind
  - terminal-state polling rejection
  - provider error and timeout mapping
  - render-plan generation after downloaded video binding
- Default frontend tests:
  - provider settings do not prefill secrets
  - video job panel shows submitted/processing/completed/failed/timeout states
  - poll action handles provider errors
  - render plan reflects bound video assets
- Demo tests:
  - Coffee Demo remains self-contained
  - additional fixtures import deterministically if added
- Opt-in real Jimeng tests:
  - skipped unless credentials and an explicit real-provider flag are present
  - submit one short controlled job
  - poll to terminal state
  - download video
  - bind asset to shot
  - export render plan
  - assert no credential values appear in responses or logs
- Manual release smoke:
  - start backend/frontend
  - import Coffee Demo
  - generate/export render plan
  - run one real Jimeng validation only if v1.1 claims real-provider support

## Release Criteria

v1.1 may release when:

- Default backend, frontend, build, lint, and smoke checks pass.
- Provider contract tests are green without real credentials.
- Real Jimeng support is either:
  - proven with an opt-in real-account smoke and documented as supported, or
  - explicitly documented as not shipped in v1.1 after feasibility review.
- Credential values are never returned by GET/PUT responses and are not prefilled in the frontend.
- Failed, timed-out, or download-failed jobs do not create misleading completed assets.
- Cost controls for real-provider tests are documented and excluded from default CI.
- README/SPEC/release notes accurately state the real Jimeng status.

## Recommended PR Sequence

1. `docs: add v1.1 provider contract and release gates`
   - Document Jimeng lifecycle states, credential trust model, timeout policy, and real-provider test gating.

2. `test: add provider contract coverage`
   - Add focused backend tests for submit/poll/download/bind/error/timeout behavior and no-secret guarantees.

3. `test: add deterministic provider workflow fixtures`
   - Add mocked job-state fixtures and, if useful, additional demo fixtures for validation and timeline/render-plan confidence.

4. `feat: harden Jimeng provider lifecycle`
   - Implement only the minimal changes needed to satisfy contract tests inside the existing provider path.

5. `test: add opt-in real Jimeng integration smoke`
   - Add skipped-by-default integration coverage guarded by explicit environment variables and cost warnings.

6. `feat: connect real Jimeng behind existing provider contract`
   - Proceed only if Milestone 2 passes. Keep the existing provider identity and avoid new API/page/table changes.

7. `docs: prepare v1.1 release notes`
   - Update README/SPEC/release notes with exact supported status, test commands, known limitations, and out-of-scope items.

## v1.1 Minimal Deliverable

The smallest useful v1.1 is:

- provider contract tests
- credential and job-state safety checks
- deterministic mock Jimeng lifecycle confidence
- documented real Jimeng feasibility result
- no new provider/API/page/table

If real Jimeng access is confirmed early, include a narrow real integration through the existing Jimeng path. If access is not confirmed, release v1.1 as provider-contract hardening and defer real Jimeng to v1.2.
